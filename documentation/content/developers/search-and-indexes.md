# Search And Indexes

**What this page achieves:** you will learn how to design search features around Anvil indexes rather than bucket scans or unsafe post-filtering.

Search is a product feature, not only an infrastructure feature. Users expect relevant results, fast filters, correct permissions, useful empty states, and stable behavior while data changes. Anvil gives developers index families that share object identity, versions, metadata, authorization, and watch-driven maintenance.

## Design search from user intent

Start with user questions:

- "Show invoices for this customer from last month."
- "Find the contract that mentions limitation of liability."
- "Show images similar to this one."
- "Find all artifacts produced by build 1842."
- "Search only documents I am allowed to see."

Each question maps to one or more index families. The application should not start with "list every object and filter locally". That approach fails at scale and can leak unauthorized data.

## Metadata query pattern

Use metadata indexes for exact or range filters:

```text
bucket: documents
prefix: tenants/acme/projects/p-123/
filters:
  document_type = contract
  status = signed
  effective_date >= 2026-01-01
sort: effective_date desc
```

The index handles field lookup and path scope. The authorization layer filters visibility before returning results.

## Full text query pattern

Use full text search when the user enters words or phrases:

```text
query: "payment terms"
fields: extracted_text, title
prefix: tenants/acme/projects/p-123/
filters: document_type = contract
snippet: true
```

Treat snippets as sensitive data. A snippet from a private document is a leak. Anvil returns snippets only for authorized results.

## Vector query pattern

Use vector search when similarity matters. A query vector may come from text, image, audio, or video embedding:

```text
embedding_model: text-embedding-v3
vector: [0.021, -0.017, ...]
metric: cosine
limit: 20
filters: language = en-GB
```

Vector indexes require stable dimension and metric settings. A vector generated by one model is not interchangeable with a vector generated by another model unless the model contract says so.

## Hybrid query pattern

Hybrid search combines text, vector, metadata, prefix, freshness, and authorization. Use it when users need both exact and semantic behavior:

```text
text: "renewal notice"
vector: embedding("contract renewal obligations")
filters:
  document_type = contract
  status != archived
prefix: tenants/acme/
ranking:
  text_weight = 0.55
  vector_weight = 0.35
  freshness_weight = 0.10
```

Do not merge unrelated result lists in application code unless you also preserve authorization, consistency, deduplication, and ranking semantics. Prefer Anvil's hybrid query model.

## Consistency choices

An index may be slightly behind recent writes. The application chooses what it needs:

| Need | Behavior |
| --- | --- |
| Fast eventual result | Return from latest published generation. |
| Read-your-write | Wait until index processed caller's mutation cursor. |
| Strict workflow gate | Require a minimum source cursor or fail as not ready. |

A UI can show a loading state when strong consistency is required. A dashboard might accept eventual freshness. A compliance export should require a proven cursor.

## Search result handling

Store and display result metadata deliberately:

- object key and version;
- content hash where relevant;
- title or display label;
- score breakdown when useful for debugging;
- authorization revision used;
- snippet provenance;
- index generation/cursor for audit paths.
